# mavpilot v0.2.0 — Production-Readiness Refactor: Design Spec

**Date:** 2026-05-28
**Author:** swarm-analysis + brainstorming session
**Status:** Approved — ready for implementation planning
**Target version:** 0.2.0 (from current 0.1.0 alpha)

---

## 1. Goal

Bring `mavpilot` from alpha (v0.1.0) to production-usable state (v0.2.0) by:
1. Fixing all safety-critical and high-priority bugs identified in the 4-agent swarm audit.
2. Decomposing the monolithic `controller.py` (1120 lines) into 8 focused submodules behind a stable facade.
3. Decomposing `_viz.html` into separate HTML/JS/CSS files using ES modules.
4. Establishing a real test architecture (unit + protocol + integration) replacing the 12-line placeholder.
5. Tightening CI, packaging, and linting.

**Non-goals (out of scope for v0.2.0):**
- PX4 SITL CI integration (placeholder marker only; full job is v0.3.0 work).
- Hardware-in-the-loop (HIL) tests.
- New flight features (waypoint missions, geofencing, etc.).
- Multi-vehicle support.
- Migration off pymavlink.

### 3.1 Key boundary: `_mav_lock`

The single hard rule that closes the worst race condition from the audit:

> **All access to `self.mav.mav.*_send` and `self.mav.recv_match` MUST go through `_connection.MAVLinkConnection`, which holds the single `_mav_lock`.**

Every other module (`_commands`, `_streamer`, `_telemetry`, `_mission`, `_safety`, `_precision_land`) receives a `MAVLinkConnection` instance via constructor injection and calls `conn.send(msg)` / `conn.recv(...)`. This makes the lock invariant enforceable by code review and makes the boundary easy to mock for tests.

### 3.2 ACK routing contract

`_commands.CommandSender` maintains:

```python
_pending_acks: dict[tuple[int, int, int], asyncio.Future]  # (cmd_id, target_sys, target_comp) → Future
```

`send_command_long(cmd_id, ..., timeout: float | None = None)` returns an awaitable that:
- if `(cmd_id, target_sys, target_comp)` already has an in-flight Future → raises `DroneError("duplicate in-flight command: <cmd_id>")` immediately. Callers needing parallel issuance of the same command must serialize externally.
- creates a Future, registers it, sends the COMMAND_LONG via `_connection.send()`
- `await asyncio.wait_for(future, timeout or DEFAULT_ACK_TIMEOUT_S)` where `DEFAULT_ACK_TIMEOUT_S = 2.0`
- raises `DroneError` if ACK result is not `MAV_RESULT_ACCEPTED` (or if timeout expires)

**IN_PROGRESS handling:** PX4 emits `MAV_RESULT_IN_PROGRESS` as an interim ACK for long-running commands (e.g., `NAV_LAND`, `NAV_TAKEOFF`). When `_telemetry.handle_command_ack(msg)` receives `IN_PROGRESS`, it does NOT resolve the Future — instead it resets the `wait_for` deadline by re-arming a timer (implementation: each Future is paired with a per-command `monotonic()` deadline; on IN_PROGRESS the deadline is pushed forward by `timeout`). The Future resolves only on a terminal result (`ACCEPTED`/`DENIED`/`UNSUPPORTED`/`FAILED`/`TEMPORARILY_REJECTED`).

**Per-command timeout overrides:** mission-level callers MUST pass explicit `timeout=` for slow commands. Recommended values:
- `set_mode`, `arm/disarm`, `param_set`: 2.0 s (default)
- `apply_safe_params` (sends ~10 params sequentially): 15.0 s total
- `NAV_LAND`, `NAV_TAKEOFF`: 30.0 s (the autopilot only sends final ACCEPTED on landing/altitude reached)
- `DO_FLIGHTTERMINATION`: 1.0 s (fast or never)

This eliminates the silent-failure mode where `apply_safe_params` etc. log `COMMAND_DENIED` but never raise, and also avoids the false-positive where `IN_PROGRESS` was treated as completion.

### 3.3 Public API changes (breaking from v0.1.0)

| Old | New | Reason |
|---|---|---|
| `drone.close()` (sync) | `await drone.aclose()` (async) + `async with DroneController(...) as drone:` | Properly cancels in-flight tasks; deterministic shutdown |
| `VizServer(port=...)` binds 0.0.0.0 | `VizServer(host="127.0.0.1", port=...)` | Stops accidental LAN exposure |
| `emergency_land()` sends termination only on exception | Chains `AUTO_LAND mode → wait 10s → MAV_CMD_NAV_LAND command → wait 5s → DO_FLIGHTTERMINATION`. Pure "land NOW" semantics — RTL is intentionally NOT in this chain because a failed AUTO_LAND signals a serious problem where RTL (requiring healthy EKF+GPS and minutes of flight) is even less likely to succeed. RTL lives in `return_to_launch()` for the orthogonal "normal mission end" scenario. | Robust against `land()` timeout without conflating two different intents |
| `return_to_launch()` returns True on disarm-in-air | Requires `landed_state == ON_GROUND` AND `not armed` AND `alt < threshold` | Correctness |
| `precision_land()` returns `bool` | Returns `PrecisionLandResult(status: PrecisionLandStatus, final_position: Position, iterations: int)` where `PrecisionLandStatus` is an `Enum` with members: `LANDED` (full descent completed via AUTO_LAND handoff with marker centered), `HANDED_OFF` (descended to `min_altitude_floor_m` with marker locked, AUTO_LAND triggered, but `landed_state == ON_GROUND` not yet observed before method returned), `ABORTED_AT_FLOOR` (descended to floor but marker lost or off-center at handoff time — AUTO_LAND NOT triggered, drone holds floor altitude), `MARKER_LOST` (marker callback returned None for `marker_lost_timeout_s` seconds before reaching floor), `TIMEOUT` (overall `timeout_s` reached). No `reason` field — status is the reason. | Caller can react to outcome with explicit, non-overlapping states |
| `precision_land(...)` no altitude floor | Adds `min_altitude_floor_m=0.3` parameter; descent stops at floor if marker is lost or off-center; only when marker is locked AND centered within `horizontal_tolerance_m` does descent proceed below floor for `AUTO_LAND` handoff | Safety |
| `DroneError` in `mavpilot.types` | `DroneError` in `mavpilot.errors`; re-exported from `mavpilot.__init__` | Conventional layout |
| CLI Ctrl-C raises `KeyboardInterrupt` mid-flight | CLI catches and triggers `aclose()` → `emergency_land()` | Safety |
| `MarkerObservation.dz` field exists but unused | Kept in dataclass with explicit `# Reserved for future vertical correction; precision_land does not read this in v0.2.0` comment. NOT removed (would be a breaking change with no benefit; field is cheap). | Avoids unnecessary breakage |

The constructor `DroneController(connection_string, ...)` gets these new kwargs (all with safe defaults to limit churn):
- `viz_host: str = "127.0.0.1"`
- `viz_max_clients: int = 32`
- `command_ack_timeout_s: float = 2.0`
- `telemetry_watchdog_s: float = 2.0`
- `min_altitude_floor_m: float = 0.3` (default for precision_land)

## 4. Threading model

| Thread | Owner | Responsibilities | Lock contract |
|---|---|---|---|
| Asyncio main | `DroneController`, `_mission`, `_precision_land`, `_commands` | User-facing methods; ACK await; mission state machine | Calls `_connection.send()` which acquires `_mav_lock` |
| Receiver | `_connection` | `recv_match()` loop, dispatches to `_telemetry.handle()` | Acquires `_mav_lock` for each recv |
| Heartbeat | `_connection` | Sends HEARTBEAT @ 1 Hz | Acquires `_mav_lock` per send |
| Streamer | `_streamer` | Sends SET_POSITION_TARGET_LOCAL_NED @ `loop_hz`; runs telemetry watchdog | Acquires `_mav_lock` per send |
| VizServer HTTP | `viz.server` | Accepts SSE connections | Independent of MAVLink |
| VizServer SSE workers | `viz.server` (per client) | Streams events to one browser | Independent; reads from per-client `queue.Queue` |
| Mock sim (optional) | `_mock` | Replaces receiver+heartbeat in mock mode; simulates dynamics | No MAVLink lock needed; pure in-process state |

`_mav_lock` is a `threading.Lock` (not asyncio) because two of its acquirers are OS threads. Asyncio callers hold the lock for the duration of a single `send` only (microseconds); no awaits happen under the lock.

**Critical timeout constraint on the receiver thread:** `recv_match(blocking=True, timeout=0.05)` — 50 ms maximum. Any longer (e.g., the 500 ms from v0.1.0) lets the receiver hold `_mav_lock` long enough to starve the streamer thread, which sends OFFBOARD setpoints at `loop_hz` (default 50 Hz = 20 ms period). A 500 ms recv timeout would block up to 25 setpoints in a row → PX4 OFFBOARD timeout (≈500 ms) fires → vehicle falls out of OFFBOARD mid-mission. 50 ms is the upper bound: at worst one setpoint is delayed, which is recoverable. This timeout is part of the lock contract and MUST be enforced in `_connection.MAVLinkConnection`.

## 5. Phased execution plan

Five phases, all landing as commits on `dev/v0.2.0`:

### Phase 1 — Safety critical (TDD for math + protocol)
1. **`cli.py` refactor**: extract signal handler logic into `handle_shutdown_signal(drone, loop) -> Awaitable[None]` (unit-testable). Add `--mock` flag (replaces ad-hoc `mock://` connection-string workaround). Ctrl-C → triggers `aclose()` → `emergency_land()` chain.
2. `emergency_land()` chain: `AUTO_LAND` mode → wait 10s for landed_state == ON_GROUND → if not landed, send `MAV_CMD_NAV_LAND` command (sometimes accepted when mode-switch is stuck) → wait 5s → if still not landed, send `DO_FLIGHTTERMINATION`. No RTL in this path (separate concern). **`emergency_land()` does NOT check the `_watchdog_tripped` flag** (introduced in Phase 2) — it's the path the watchdog is meant to trigger; ignoring it would defeat the purpose.
3. `_mav_lock` introduced as module-level inside controller.py (will move to `_connection.py` in Phase 3, but contract starts now). `recv_match` timeout shortened to 50 ms (see §4 constraint).
4. `pixel_to_body_offset` fixed pinhole math (TDD: failing test at `px=0.5, hfov=90°, alt=10` → expected `dy=5.0`, current returns `4.14`)
5. `precision_land`: latched `descend_floor_z`, horizontal gate before `AUTO_LAND` handoff, `PrecisionLandResult` return type with `PrecisionLandStatus` enum (LANDED / HANDED_OFF / ABORTED_AT_FLOOR / MARKER_LOST / TIMEOUT).
6. `viz.publish()`: NaN/Inf sanitizer (NaN→None) + `json.dumps(allow_nan=False)` wrapped in try/except with log
7. `VizServer` default host → `127.0.0.1`; `host` parameter exposed
8. `return_to_launch()` correctness: landed_state==ON_GROUND AND not armed (not OR)

### Phase 2 — Protocol & state hygiene
9. COMMAND_ACK as `asyncio.Future` per `(cmd_id, sys, comp)` (TDD with FakeMavConnection)
10. `wait_until_ready` adds EKF/GPS health check via `SYS_STATUS.onboard_control_sensors_health` (`AHRS` bit and `GPS` bit when applicable)
11. arm-then-OFFBOARD reordering in `takeoff()`: stream → arm → set_mode(OFFBOARD)
12. Streamer telemetry watchdog: if `monotonic() - last_local_pos_ts > telemetry_watchdog_s`, set `_watchdog_tripped` flag and log; streamer continues publishing last-known setpoint to keep OFFBOARD alive; **next call to any high-level mission method** (`takeoff`/`goto`/`land`/etc.) checks the flag and raises `DroneError("telemetry lost")`. CLI's `except` then triggers `emergency_land()` chain. Watchdog does NOT directly command flight modes — it surfaces the error to the orchestration layer.
13. `time.monotonic()`-based `time_boot_ms` (with process-start offset)
14. `srcSystem` filtering in all telemetry handlers (filter to `target_system`)
15. `asyncio.get_event_loop()` → `asyncio.get_running_loop()`
16. `get_yaw_deg()` normalized to `[-180, 180]`
17. `recv_match` under `_mav_lock`

### Phase 3 — Python decomposition
18. Extract `_connection.py` with `MAVLinkConnection` class (owns `_mav_lock`, receiver thread, heartbeat thread, connect/close)
19. Extract `_telemetry.py` with `Telemetry` (state dict + handlers); ATTITUDE/LOCAL_POSITION_NED/HEARTBEAT/COMMAND_ACK/STATUSTEXT/SYS_STATUS handlers
20. Extract `_commands.py` with `CommandSender` (ACK futures, set_mode, arm/disarm, param_set with bit reinterpret)
21. Extract `_streamer.py` with `OffboardStreamer` (setpoint dataclass, slewing yaw, watchdog)
22. Extract `_mission.py` with `MissionOps` (takeoff, goto, goto_relative, goto_body_relative, set_yaw, hover, land, rtl, emergency_land)
23. Extract `_precision_land.py` with `PrecisionLand`
24. Extract `_safety.py` with `apply_safe_params`, `wait_until_ready`
25. Extract `_mock.py` with `MockMavConnection` + sim thread
26. Move `DroneError` to `errors.py`; re-export from `__init__`
27. `DroneController` becomes facade (~250 lines) composing all of the above; implements `__aenter__/__aexit__`
28. Load viz static via `importlib.resources.files("mavpilot.viz") / "static"`

### Phase 4 — Viz decomposition
29. Create `mavpilot/viz/static/{index.html, main.js, sse.js, scene.js, telemetry.js, log.js, styles.css}`
30. `index.html` uses `<script type="module" src="./main.js">`
31. `main.js` imports from `sse.js` / `scene.js` / `telemetry.js` / `log.js`
32. `VizServer` serves the directory with correct MIME types (`.js`→`application/javascript`, `.css`→`text/css`, `.html`→`text/html`)
33. SSE shutdown sentinel: `stop()` enqueues `None` to all client queues; worker loop breaks on `None` (removes ≤15 s shutdown tail)
34. Cap on concurrent SSE clients via `max_clients` parameter (default 32); excess connections rejected with 503

### Phase 5 — CI / packaging / quality

## 7. Git workflow

1. Branch off current `main` (43d51d6) as `dev/v0.2.0`
2. Each phase = 1–N commits on `dev/v0.2.0`, with conventional-commit prefixes:
   - `fix: …` for safety/protocol fixes
   - `refactor: …` for decomposition
   - `test: …` for test additions
   - `docs: …` for README/CHANGELOG updates
   - `chore: …` for packaging/CI
3. README (RU + EN) is updated within each phase via a dedicated `docs: …` commit at the end of that phase
4. After all phases pass full test suite (`pytest -v`): final commit bumps version + CHANGELOG
5. Tag `v0.2.0` is created **on `main`** after merge (not on `dev/v0.2.0`)
6. After merge and tag, `dev/v0.2.0` may be kept for archive or deleted

## 9. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Decomposition (Phase 3) introduces subtle behavior changes | Medium | High (silent flight bugs) | Integration tests written in Phase 1-2 must still pass after Phase 3 with zero modifications |
| `_mav_lock` causes performance regression at high `loop_hz` (e.g. 250 Hz) | Low | Medium | Lock held only for the `*_send` call (~µs); not under await; benchmark in `test_streamer.py` |
| mypy unmask in Phase 5 reveals many real bugs | Medium | Medium (cleanup work) | Allow Phase 5 to span more commits if needed; do not block release on getting mypy fully green if findings are non-safety-critical |
| Viz ES-modules don't work in older browsers (e.g. Chromium 70-) | Low | Low | Library is developer-facing; require modern browser; document in README |
| pymavlink `>=2.4,<3` excludes some installed versions | Low | Low | Verify on PyPI the version distribution; relax if needed |

## 10. Acceptance criteria

`v0.2.0` is ready to tag when **all** of:
1. All Phase 1 and Phase 2 fixes implemented with passing tests
2. All decomposition (Phase 3, 4) completed; `controller.py` ≤ 300 lines
3. `pytest tests/` green on Python 3.10, 3.11, 3.12 (3.13 informational)
4. `ruff check mavpilot/` green with new rule set
5. `mypy mavpilot/` green (or remaining errors documented as acceptable)
6. `black --check mavpilot/ tests/` green
7. Coverage ≥ targets in §6.4
8. README RU + EN reflect new API; `CHANGELOG.md` lists all breaking changes
9. Manual sanity check: `python -m mavpilot --mock --precision-land` runs end-to-end without error (the `--mock` flag is added in Phase 1, replacing the v0.1.0 `mock://` connection-string workaround)
10. Manual sanity check: viz UI loads in browser (default `http://127.0.0.1:8765`) and shows live telemetry; Ctrl-C cleanly closes the process
